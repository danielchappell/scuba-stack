# Build the Lever

Some changes solve one problem. A few change the structure so a whole category of future problems stops existing. When you can see one of those, it is usually worth more than the immediate fix.

## Look for the change that makes ten things trivial

The lever is the point where one structural improvement (a better abstraction at the right seam, a fixed root cause, a tool that automates the repetitive step) makes many downstream tasks easy or unnecessary. Solve the case, then ask what would have made this case not exist.

## The lever is found, not invented

Leverage comes from a real pattern in the work, not from cleverness imposed on it. If you cannot point at the recurring cost it removes, you are probably building speculative machinery, not a lever. Hold this against subtract-before-you-add: don't build leverage nobody needs.

## Spend effort where it compounds

Given a choice between a fix that helps once and a comparable effort that helps every future instance, prefer the second, when the future instances are real. The test is whether the cost it removes is concrete and recurring.
